# connect_rust example

Carries protobuf messages — the same types a [connect-rust](https://github.com/anthropics/connect-rust)
service would use, generated by the [`buffa`](https://docs.rs/buffa) runtime —
over the in-process `CircularBus`:

```
Client         -> GetAccountRequest  -> AccountService
AccountService -> GetAccountResponse -> Client
```

Run it:

```bash
cargo run --example connect_rust
```

## How the message traits are wired

`messages.rs` implements the bus traits on the buffa-generated types:

- `ExtendedMessage::get_size` returns `encoded_len()` and `write_into` calls
  `encode` straight into the bus slot (`&mut [u8]` is a `buffa::bytes::BufMut`).
- `deserialize_from` calls `decode_from_slice` on the buffer. The bus header
  records the exact (unpadded) payload length, so `read` hands back precisely
  the protobuf bytes — no length prefix or framing needed, even though the
  underlying slot is padded for alignment.

## Regenerating the protobuf code

`generated/account.rs` is vendored so the example builds with only `buffa` as a
dependency (no `protoc` at build time). It was generated from
`proto/account.proto` with [`buffa-build`](https://docs.rs/buffa-build). To
regenerate after editing the proto, run this in a throwaway crate (it needs
`protoc` on `PATH`):

```rust
// build.rs
fn main() {
    buffa_build::Config::new()
        .files(&["proto/account.proto"])
        .includes(&["proto"])
        .compile()
        .unwrap();
}
```

then copy the resulting `OUT_DIR/account.rs` over `generated/account.rs`.
