# connect_rust example

Carries protobuf messages — the same types a [connect-rust](https://github.com/anthropics/connect-rust)
service would use, generated by the [`buffa`](https://docs.rs/buffa) runtime —
over the in-process `CircularBus`, decoding them **zero-copy** on the read side:

```
Client         -> GetAccountRequest  -> AccountService
AccountService -> GetAccountResponse -> Client
```

Run it:

```bash
cargo run --example connect_rust
```

## Zero-copy reads via buffa views

Senders write **owned** messages (`GetAccountResponse`), which `ExtendedMessage`
serializes into the bus slot with buffa's `encode`. Receivers are routed the
**borrowed view** types (`GetAccountResponseView<'a>`): the `Messenger!` route
names `...View`, so each handler decodes the slot in place with buffa's
`decode_view`. String/bytes fields borrow directly from the bus buffer — e.g.
`GetAccountResponseView::name` is a `&'a str` pointing into the ring slot, with
no allocation. The `'a` lifetime ties the view to the slot, so it stays valid
for the duration of the handler call (the bus guarantees the slice is live
while the handler runs).

This is zero-*allocation* decode: buffa still parses the protobuf wire format,
it just doesn't copy `string`/`bytes` fields out. It is **not** the bus's
zero-*parse* mode (raw `*const u8 -> &Message` cast); protobuf cannot use that,
since its wire format is not a memory image of the struct (see the crate
README's safety section).

### What it took to route a view to a handler

No macro changes — the view types just implement the normal traits
(`messages.rs`):

- `impl<'a> Message for GetAccountResponseView<'a>` with the **same** `ID` as
  the owned type, so routing matches the bytes the owned type wrote.
- `GetAccountResponseView::deserialize_from(&'a [u8])` calling `decode_view`.
- `impl<'a> Handle<GetAccountResponseView<'a>> for Client`.

The `Messenger!` route already carries the slot's lifetime on its `buffer`
argument, so `<View>::deserialize_from(buffer)` infers `View<'a>` and the
`Handle<View<'a>>` impl matches.

## Regenerating the protobuf code

`generated/` is vendored so the example builds with only `buffa` as a
dependency (no `protoc` at build time). It was generated from
`proto/account.proto` with [`buffa-build`](https://docs.rs/buffa-build) and view
generation enabled. To regenerate after editing the proto, run this in a
throwaway crate (it needs `protoc` on `PATH`):

```rust
// build.rs
fn main() {
    buffa_build::Config::new()
        .files(&["proto/account.proto"])
        .includes(&["proto"])
        .generate_views(true)             // emit the borrowed view types
        .generate_json(false)             // not needed here
        .gate_impls_on_crate_features(false)
        .compile()
        .unwrap();
}
```

then copy the resulting `OUT_DIR/account.rs`, `account.__view.rs`, and
`account.mod.rs` over the files in `generated/`. Include `generated/account.mod.rs`
— it wires together the owned types and the `__buffa::view` module.
